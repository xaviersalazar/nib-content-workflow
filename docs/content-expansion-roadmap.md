# TIL Content Workflow

# Content Expansion Roadmap

---

# Purpose

A forward-looking plan for growing the content library **after** the initial
launch set is fully drafted.

This is a planning document, not a production queue. Nothing here should be
scraped or drafted until the current launch backlog (the `pending` rows in
`source-registry/sources.csv`) is complete.

The goal is:

Grow breadth deliberately.

Keep trust high.

Avoid running dry on the Categories and Collections screens.

---

# Where We Stand

Snapshot at time of writing:

- **35 categories**
- **~364 topics**, most at 3–5 facts each
- **Space + Astronomy** scraped and complete; everything else `pending`
- **~78% of topics source from Britannica** (283 of 364)

At one fact per day, the launch set alone is **3–5 years** of daily content.

The takeaway: expansion is **not** about avoiding a content shortage. It is about
three things —

1. **Breadth** — the Categories and Collections screens should feel alive and
   keep growing.
2. **Seasonal relevance** — timely drops that match the calendar.
3. **Marketing beats** — a recurring "new this month" story for retention.

# Two Equally Important Levers

Expansion is not just "add more categories." There are two levers, and the
second is the one most often neglected.

## Lever 1 — Expand Existing Categories

Most categories launched with exactly 10 topics. Many can comfortably double.

Adding topics to a category the user already browses is **lower risk and lower
effort** than standing up a new category: the audience is proven, the source
institutions are already validated, and no new app-side category metadata
(color token, icon, description) is required.

Deepen before you widen. Examples:

| Category | Add topics like |
| --- | --- |
| Animals | Bats, Foxes, Owls, Frogs, Komodo Dragons, Hummingbirds, Chameleons |
| Weather | Fog, Hail, Rainbows, Floods, Droughts, Monsoons, Wind |
| Space | Solar Eclipses, Meteor Showers, Space Junk, The Kuiper Belt, Auroras |
| Human Body | Liver, Kidneys, Nervous System, Hormones, Teeth, Hair, Hearing |
| Food | Eggs, Tea, Sugar, Butter, Vinegar, Fermentation, Mushrooms |
| Sports | Cycling, Boxing, Surfing, Climbing, Gymnastics, Cricket, Rugby |
| Mathematics | Statistics, Topology, Knot Theory, Game Theory, Symmetry, Codes |
| History | Renaissance Italy, Ancient Egypt's New Kingdom, The Plague, Exploration of Antarctica |

Rule of thumb: when a category proves popular in analytics, the **first**
response is to deepen it, not to launch something new.

## Lever 2 — Add New Categories

New categories fill genuine gaps in the taxonomy. See the waves below.

A healthy month often pairs the two: one **new** marquee category plus a few
**added** topics to an existing one.

# Source Diversification Goal

> **Target: reduce Britannica from ~78% of topics to ~50% over Year 1.**

Britannica is an excellent Tier-1 source and should remain the backbone. But ~78%
concentration is a single point of failure:

- **Operational risk** — one site changing structure or blocking scraping stalls
  most of the pipeline.
- **Tone sameness** — every fact reading the same hurts the "premium, curated"
  feel.
- **Trust depth** — domain-specific institutions (NASA, NOAA, USGS, AMNH, The
  Met) carry more authority on their subject than a general encyclopedia.

How to get there without lowering quality:

1. **Every new category onboards at least one new institution.** The Wave tables
   above already assign domain-specific sources for this reason.
2. **When expanding existing categories, prefer a specialist source** over
   reaching for Britannica again (e.g. a new Animals topic → WWF, San Diego Zoo,
   or Cornell Lab rather than Britannica).
3. **Validate scraping before committing rows.** Some institutional pages are
   JS-rendered or block crawlers — the known failure mode in
   `docs/trusted-source-workflow.md`. Run `pnpm scrape:source` against a candidate
   URL first; only add the row if it returns clean markdown.
4. **Track the ratio.** Re-check the Britannica share each quarter and let it
   steer source choices for the next batch.

Quick check of the current ratio:

```bash
total=$(tail -n +2 source-registry/sources.csv | grep -vc '^$')
brit=$(tail -n +2 source-registry/sources.csv | grep -v '^$' | awk -F, '$3=="Britannica"' | wc -l)
echo "Britannica: $brit / $total"
```

# Rollout Cadence

The right rhythm for a solo, human-in-the-loop pipeline is:

> **One marquee drop + one seasonal Collection per month.**

Each drop doubles as a Collections-screen pack *and* a marketing beat. Tie drops
to the calendar so they feel timely. The schedule below assumes a mid-2026
launch — shift the months to match the real launch date.

| Month | Marquee drop | Seasonal Collection / hook |
| --- | --- | --- |
| Jul | Space Exploration | Apollo 11 anniversary (Jul 20) — "We Went to the Moon" |
| Aug | Birds + Bugs | Summer nature — "Backyard Wonders" |
| Sep | Mathematics / Languages deepening | Back-to-school — "How Your Brain Learns" |
| Oct | Mysteries + Myths expansion | Halloween — "Spooky Science" (Bats, Spiders, Venom) |
| Nov | Dinosaurs & Prehistoric Life | Crowd-pleaser marquee |
| Dec | Holidays & Traditions | Winter / Festive — Snow, Aurora, world holidays |
| Jan | Money & Personal Finance | New Year — "Fresh Start" + Habits |
| Feb | Art + Photography | Valentine's — "Art & Love" |
| Mar | Plants & Botany + Geology | Spring — "Things That Grow" |
| Apr | Environment & Ecology | Earth Day — Oceans deepening |
| May | Games & Puzzles + Computing | "Play & Logic" |
| Jun | New marquee + Year-1 recap | Launch anniversary |

## Production Math

Sustaining roughly one new category per month means **~30–50 validated
facts/month** (10 topics x 3–5 facts).

- The bottleneck is the **human review pass**, not scraping. Keep drops to one
  marquee category, not three.
- Stage rows about a month ahead, batch-scrape, then draft and review in weekly
  sittings.

---

# Suggested Workflow Addition: a `backlog` status

The `pending` status is consumed by `scrape:next` and `scrape:batch`. To stage
future rows in `source-registry/sources.csv` without them being scraped
prematurely, use a distinct status:

- `backlog` — decided on, URL recorded, **not** yet ready to scrape
- `pending` — ready; will be picked up by `scrape:batch`
- `complete` — scraped

When a drop's month arrives, flip its rows from `backlog` to `pending` and run
the batch. The batch scripts already ignore anything that isn't `pending`, so no
code change is required to adopt this convention.

# How To Use This Document

1. Finish the current launch backlog first.
2. Each month, pick **one** marquee item (new category or a split) and **one**
   expansion or seasonal Collection.
3. Prefer a non-Britannica specialist source for every new row; validate it
   scrapes cleanly before adding it.
4. Re-check the Britannica ratio quarterly and steer toward the 50% target.
5. Revisit and reprioritize the waves as analytics reveal which categories
   users actually explore.
